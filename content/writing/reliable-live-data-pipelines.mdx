A live sports product does not have one tidy database update. Fixtures move, events arrive in bursts, corrections appear later, and several product surfaces expect a consistent answer at the same time. In LetLetMe, that answer supports analytics, tournament workflows, notifications and companion experiences used by 800+ users.

The useful engineering question is not simply “how do we fetch the latest data?” It is: **how do we know what is fresh, what is authoritative and how to recover when part of the pipeline fails?**

## Separate collection from product reads

The first boundary is between external data collection and user-facing requests. Calling an upstream provider directly from a page route couples product latency and availability to a service outside the product’s control.

LetLetMe instead treats ingestion as its own workflow:

1. Scheduled jobs fetch the required source payloads.
2. Validation rejects structurally incomplete responses.
3. Normalization maps provider-specific data into domain records.
4. Scoring and derived-data steps apply shared business rules.
5. Synchronization updates relational storage and Redis.
6. Product APIs expose controlled state to clients.

This boundary lets product routes remain predictable even when an external source is slow. It also gives ingestion failures an observable place to live instead of turning them into unexplained page errors.

## Give each storage layer a clear job

Relational storage and Redis are not interchangeable copies. Relational records remain the durable source for domain state and history. Redis supports freshness-sensitive reads where rebuilding the same view repeatedly would add unnecessary work.

That distinction affects implementation:

- cache keys should describe a domain view rather than an arbitrary page;
- expiry should follow the expected update rhythm of the data;
- writes should make their ordering explicit;
- a missing cache entry should be recoverable from durable records;
- clients should not contain a second version of scoring rules.

The point is not to cache everything. It is to make the read path intentional.

## Design for partial completion

Distributed workflows rarely fail as a single clean transaction. A fetch may succeed while normalization fails. Database writes may complete before a cache refresh. A notification step may be delayed after the product state is already correct.

The operating model therefore needs more than retries. LetLetMe uses explicit verification and repair paths: check the source-of-truth record, compare derived or cached state, rebuild the affected view, and rerun only the safe stage when necessary.

This is a more useful recovery sequence than clearing an entire cache and hoping the next scheduled job repairs everything. It narrows the blast radius and makes the response explainable.

## Keep shared rules behind APIs

The platform has more than one client surface. Web views, companion clients and notification services consume overlapping state, but they should not each calculate it differently.

Shared backend rules give those clients a stable contract. REST endpoints handle focused product operations; GraphQL supports composed analytics reads; background services own ingestion and derived-data work. The exact interface matters less than keeping domain calculations in one accountable place.

## What I would preserve in another live-data product

The reusable lessons are straightforward:

- define freshness as a product requirement, not an incidental cache setting;
- keep ingestion independent from interactive requests;
- state which layer is authoritative for each kind of data;
- make partial progress and reruns observable;
- build repair operations before the first production incident;
- centralize business rules used by multiple clients.

Reliability in this kind of product comes from visible boundaries. Once collection, normalization, durable state, live reads and recovery are separate concepts, the system becomes much easier to reason about and operate.
